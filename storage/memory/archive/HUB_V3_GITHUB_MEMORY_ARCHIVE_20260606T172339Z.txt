## Executive verdict

The current architecture is correct in principle:
- 8 buttons = 8 independent route-blocks.
- Each route has its own route_id, callback namespace, case_namespace, screens, agents, reports, schemas, links, safety locks.
- The blocks are separate but connected through explicit LINKS, not chaotic shared flow.
- Route 07 is the decision/paused/block dispatcher.
- Route 08 is system/memory/rules/security, not daily product work.
- Route 06 is the most detailed and already has a deeper code draft.

However, before server transfer, one more GitHub-memory restructure layer is required:

Reason: current code drafts are good memory drafts, but not yet strict enough as server-ready contracts. They need unified route contracts, validation rules, callback length checks, state machine, shared schemas, and self-review requirements.

## External / internet audit findings to apply

1. Telegram Bot API callback_data must be 1-64 bytes. All V7.3C callbacks must be validated before server transfer.
2. Telegram callback_query must be handled as an update type; callback handlers must acknowledge/respond cleanly.
3. If aiogram-style routing is used, independent routers/modules can be attached through include_router/include_routers. This supports the 8 independent route-block design.
4. Nested routers must not form circular includes. Route modules must never include each other directly.
5. If python-telegram-bot ConversationHandler-style state machines are used, sequential update processing/state consistency must be respected. Route state must not be corrupted by parallel flow.
6. eBay Inventory API has separate inventory item, offer, inventory item group, business policies, and publish requirements. Therefore 01/02/03 publish gates must remain separate from dry route logic; variants route 02 must account for inventory item group later, but not now.

## Gaps found before server transfer

GAP 1 — Need strict route contract file
Current drafts use a pattern, but need one explicit V7.3C route contract defining required fields for every module:
ROUTE_ID, ROUTE_TITLE, CASE_NAMESPACE, CALLBACKS, BUTTONS, SAFETY_LOCKS, SCREENS, AGENTS, REPORTS, SCHEMAS, LINKS, get_home_screen, get_callback_response, self_review.

GAP 2 — Need callback validator
All callbacks must be checked:
- unique globally;
- 1-64 bytes;
- start with v73c:;
- route-specific callbacks match r01/r02/r03/r04/r05/r06/r07/r08;
- no duplicate callback across routes.

GAP 3 — Need route state model
Each route needs state fields:
case_id, route_id, status, current_screen, previous_screen, created_utc, updated_utc, operator_id, pending_decision, links_to_other_routes, safety_lock_state.

Status values:
NEW, COLLECTING, IN_PROGRESS, NEEDS_REVIEW, READY_FOR_GATE, PAUSED, BLOCKED, CANCELLED, DONE.

GAP 4 — Need cross-route handoff contract
Links must not be only text. Need handoff objects:
from_route, to_route, case_id, reason, payload_ref, decision_required, created_utc, allowed_actions.

GAP 5 — Need route 07 as real inbox aggregator
Route 07 must receive PAUSED/BLOCKED/DECISION_REQUIRED from all routes through decision_card schema.

GAP 6 — Need route 08 as read-only registry
Route 08 must read route registry, memory pointers, archive index, rules, version, and security locks. It must not mutate routes without approval.

GAP 7 — Need dry self-review output for every route
Each candidate module must produce self_review with:
unique callbacks, callback byte limit, all safety locks false, required fields present, required global buttons present, no live actions, no secrets.

GAP 8 — Need server transfer packet to be generated from V3 contract, not from loose drafts.
The next server packet must materialize strict files only after V3 contract is saved.
