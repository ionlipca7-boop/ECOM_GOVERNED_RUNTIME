Reason:
Telegram buttons and route-blocks are not just UI. They must map to marketplace-safe actions, product compliance, listing gates, finance/stock, and platform-specific adapters.

## Official source audit applied

Telegram:
- callback_data must be 1-64 bytes.
- callback_query is a real update type.

EU/Germany product compliance:
- GPSR Regulation (EU) 2023/988 is in force and applies to product safety in EU market.
- Online selling needs product safety/compliance information and risk-aware product handling.

Ebay:
- Inventory API separates Location, Inventory Item, Offer, Inventory Item Group, business policies and publish.
- Inventory Item needs unique SKU.
- Offer becomes live listing only after publish.
- Multiple-variation listings use Inventory Item Group and publishOfferByInventoryItemGroup.
- Offers require payment, fulfillment and return business policies.

Amazon:
- SP-API has Listings Items, Listings Restrictions and Product Type Definitions.
- Amazon requires product-type/category-driven schema handling; listings are not simply eBay-like offers.

Kaufland:
- Seller API covers product data, variant suggestions, inventory, orders, invoices, tickets, returns, reports, push notifications and CSV interfaces.
- Kaufland supports API and CSV workflows; integration must not assume only one API style.

OTTO:
- OTTO Market API distinguishes Sellers and Service Partners.
- APIs cover business domains such as orders and products.
- OpenAPI specifications and changelog must be used for integration.

## Platform-specific adapters

### eBay adapter
Entities:
- inventory_location
- inventory_item
- offer
- inventory_item_group for variants
- business_policies payment/fulfillment/return
- listing_id after publish

Special rules:
- SKU uniqueness required.
- Business policies required.
- Variant route 02 must later map to inventory item group.
- Publish action is separate from create/update offer.
- Revise/update must stay through Inventory API if listing created by Inventory API.

Route impact:
01 simple item -> inventory_item + offer + publish gate.
02 variants -> inventory_items + inventory_item_group + offers + group publish gate.
03 listing studio -> update offer/listing only through eBay update gate.
04 lifecycle -> read metrics/listing health only until live gate.
05 stock -> quantity update gate separate.

### Amazon adapter
Entities:
- marketplace_id
- SKU
- product_type
- listings item
- product type definition schema
- listing restrictions
- offers/pricing/fulfillment may require separate APIs.

Special rules:
- Product type/category schema is central.
- Restrictions must be checked before listing.
- ASIN/catalog matching may be required.
- Listing content fields differ from eBay.

Route impact:
01/02 must not assume eBay category/specifics structure.
03 listing studio needs Amazon buyer/search fields separately.
04 lifecycle needs Amazon metric/report adapter.
06 radar can use Amazon reviews/search as proof but no Amazon write without future gate.

### Kaufland adapter
Entities:
- product data
- variant suggestions
- inventory/units
- orders
- invoices
- returns
- reports
- CSV/API interface.

Special rules:
- May use API or CSV.
- Product data and inventory can be separate workflows.
- Variant suggestions may be a separate process, not same as eBay inventory item group.

Route impact:
02 variant logic must be adapter-specific.
05 stock/finance must support Kaufland order/invoice/return data.
04 lifecycle should consume reports when available.

### OTTO adapter
Entities:
- seller/service partner context
- products
- orders
- API specs via OpenAPI
- changelog/version tracking.

Special rules:
- Integration must track OTTO API version/changelog.
- Do not assume eBay/Amazon listing fields.
- Future OTTO support should be adapter module, not core route rewrite.

Route impact:
08 version/rules must track OTTO adapter version later.
07 decision center must show OTTO-specific blockers separately.

### Etsy future adapter
Status: FUTURE_ONLY
Rule:
Do not build Etsy server logic now. Store as possible future adapter because product type, handmade/vintage/supply policies, listing taxonomy and inventory/variation model differ from eBay/Amazon/Kaufland.
